# Low Level Action Diagram

A low-level action diagram is a detailed breakdown of how your system actually carries out a specific function — showing not just what needs to happen, but *which part of the system is responsible for each step*, and *what triggers the next step to begin*.

Think of it this way: in Stage 1, you created a high-level action diagram that described your system's behaviour from a bird's-eye view — broad actions like "detect the command to release" or "eject the contents of the receptacle."

At that point, you didn't yet know enough about your design to go deeper.

Now that you have identified your subsystems, you can open up each of those high-level action blocks and show the finer sequence of events happening inside.

## **2. Why This Deliverable Matters**

The key purpose of this deliverable is to help you think rigorously and clearly about the behaviour of your system at a level of detail where real design decisions begin.

Every action you capture here will eventually become a testable requirement that your design must satisfy.

## **3. Step-by-Step Instructions**

> **Remember!** The goal of the action diagram is to help you think clearly about what needs to happen, not how it will be implemented. Keep language functional and abstract.

#### **3a. Review High-Level Action Diagrams**

Before decomposing, review the high-level action diagrams and the functional requirements from Stage 1 against the priority use case behaviors. Verify that each high-level action block still correctly represents the behavior of the system at the current stage of understanding. Correct and extend the high-level diagrams as needed.

Depending on the nature and complexity of your system, you may require diagrams that go to 3, 4 or maybe more decomposition levels. Some may suffice with 1 or 2 decompositions. **Use your understanding and practices from programming to make a decision.** Similar to how you would make a decision on whether something should be in the main() function or need a function of its own. As said earlier, the goal is to help understand the system.

Prioritize the use cases: focus decomposition effort on the use cases with the greatest functional complexity, the highest operational risk, or the most cross-subsystem activity.

For each action block being decomposed, consider these steps:

1.  Label parallel action streams with the performing subsystem (or asset).

    - Each stream represents one subsystem's thread of behavior within the scenario.

    - You can use the “Branch Asset” entity by drag-and-dropping it to the concerned branch.

    - By naming this as your performing asset, Innoslate will create an asset entity of that name and assign it a “performs” relationship to the actions in the branch.

    - Alternatively, you can create assets separately and assign them relationships.

> *Note that subsystems created after this step do not have the System in the “decomposes” relationship. This will be explicitly done when decomposing the system using the Physical I/O diagram.*

2.  Use inputs/outputs entities to connect action streams.

    - These are triggers that pass control between action streams.

3.  Model materials and physical flow using “Resources”

    - Use “Resources” to model raw materials, semi-finished, or finished goods that can be generated by actions or used by actions.

    - You can assign initial, minimum, and maximum amounts to resources to model how a real-world process flow might work.

4.  Use Loops for repeated operations

    - Use Loops for recurring operations, think of the logic in the following way:  
      for \<Loop Statement\> = true  
      continue  
      Else exit

    - You can use the script feature in the top ribbon to program conditions for loops or set a count of iterations. You can learn more about them [<u>here</u>](https://help.specinnovations.com/simulation-script-guide), though these are advanced features and you are not necessarily expected to use them.

5.  Use OR branches for decision-making

    - Use OR for situations where your system has to take a decision for future actions.

    - In the example case below, the system has to make a decision where one path will consume more time and resources, but that action may be required to meet a user's need.  
      ![](../../assets/images/image19.png){ style="width:2.98438in;height:2.11822in" }

6.  Add duration to actions (for analysis later)

    - For every un-decomposed action unit, enter what time that action will take in the duration option in the attributes section.

    - This will enable you to perform a time-motion study to analyse bottlenecks in your system.

7.  Maintain hierarchical numbering

    - Number the decomposed actions hierarchically (e.g., action 1.4 decomposes into 1.4.1, 1.4.2, etc.) to maintain traceability to the parent diagram.

8.  Keep descriptions functional, not implementation-specific

    - Keep action descriptions functional: use "the system shall detect the release" rather than "the sensor reads the voltage."

    - Implementation language belongs in design documentation, not in action diagrams.

***Note:** Do not limit decomposition diagrams to the nominal (happy-path) scenario. Non-ideal paths — failures, timeouts, user errors — are often where the most important functional requirements live. Capturing them in the action diagrams ensures they enter the requirements baseline rather than being discovered during integration.*
