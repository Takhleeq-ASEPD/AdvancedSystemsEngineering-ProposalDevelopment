# List of Stakeholders

**What this deliverable is**

A list of stakeholders or entities that have an **interest** in the **outcome** of the project or problem.

**Why this deliverable is important**

A project rarely **fails** because engineers couldn’t build it.

It fails because it **solved the wrong problem** for the wrong people.

Identifying stakeholders early prevents **requirements defects**, and requirements defects are the most expensive mistakes you can make.

Stakeholders generate real requirements. If you skip stakeholder identification, you build features instead of solving problems.

**Guidelines**

2.  Organize the stakeholders in a *Power vs Interest Matrix*.

    - Power refers to the stakeholder’s authority to influence decisions or direct the project.

    - Interest refers to the degree to which the stakeholder is affected by the system or its outcomes.

> This exercise is to help you prioritize stakeholders and determine how much engagement, communication or consideration each of the stakeholders need.

- Stakeholders with **high power/influence** and **high interest** are **key players**. These should be involved in governance/decision making.

- Stakeholders with **high power/influence** and **low interest** are the ones you should look to increase their interests.

- Stakeholders with **low power/influence** and **high interest** are the ones you should consult on areas of interest.

- Stakeholders with **low power** and **low interest** are the **least important stakeholders**.

These artifacts represent **what was observed or stated**, *not conclusions*. No interpretation, no filtering, no conclusions at this stage.

### **Why this deliverable is important**

- It prevents user needs from being invented, summarized loosely, or remembered incorrectly.

- It creates **auditability**: reviewers can see *exactly* where a need came from.

- It enables strong traceability inside Innoslate.

- It enforces professional engineering practice: **claims must have evidence**.

If raw data is missing, user needs are **not valid**, regardless of how well they are written.

### **Upload all Raw Evidence in the official folder**

There is an official folder for your project assigned to you by the Takhleeq office.

- Make sure the online link to that official folder is visible on the dashboard of your Innoslate project.

- All raw evidence files (screenshots, images, audio/video recordings, pdfs, etc.) need to be uploaded in this folder.

Below we outline the steps to organize the files inside the official folder:

1.  Create separate folders for each stakeholder. Use role-based stakeholders names (no personal names) for each folder.

2.  Inside each stakeholder folder, during uploading a file, name the file with the convention “{file type}\_{Context}\_{YYYY-MM-DD}.{extension}”. For example, “INT_Preventive_Maintenance_2026-01-14.mp3”.

For file type, use the following prefixes:

| **Type**                         | **Prefix** |
|----------------------------------|------------|
| Interview recording              | INT        |
| Meeting minutes                  | MIN        |
| Email acknowledgement screenshot | ACK        |
| Photo/Image                      | IMG        |
| Video recording                  | VID        |
| Audio recording                  | AUD        |
| Reference document               | REF        |
| Measurement data                 | DAT        |

### **Uploading Raw Evidence in Innoslate**

For each raw evidence, you are required to create two entities in Innoslate:

1.  An artifact where we upload the relevant file (screenshot, audio/video recording, image, pdf, etc.)

2.  A document which contains numbered statements transcribing the contents of the file uploaded in the artifact (minutes in the screenshot, conversation in the recording, relevant extracts in the pdf, etc.)

Below we outline the instructions specific to each type of evidence:

1.  For **meeting minutes** and **interview notes**,

    1.  <u>Creating the artifact</u>

        1.  Get meeting minutes/interview notes acknowledged by the stakeholder over email. If the meeting was with a low-skilled labor who cannot read/write, this step can be skipped.

        2.  Take a screenshot of the meeting minutes along with the acknowledgement.

        3.  On Innoslate, go to “Database” view, click ![](../../assets/images/image16.png){ style="width:0.82672in;height:0.23148in" }, and choose Artifact.

        4.  Enter a meaningful name (e.g., Factory Visit – Maintenance Supervisor – 12 Jan) which includes date, stakeholder role, and context of data collection.

        5.  Name the artifact as “ART.n” where n is an integer

        6.  Upload screenshot of the acknowledgement to the artifact created for meeting minutes/interview notes.

    2.  <u>Creating the document</u>

        1.  Go to the “Documents” view and create a “Notes Document”. Name it “Extracts from {name of the artifact in step a}”. Number it “EX.n”, where n is an integer. Keep the template as “Blank Template”.

        2.  Add a description containing

            1.  Context (meeting, site visit, email, etc.)

        3.  Create a relationship between the document and the artifact using “related to”.

## **3. User Needs Document**

### **What this deliverable is**

The **User Needs Document** captures *what users actually need* from the system, expressed in their language, **before** any solution or design decisions are made.  
Each user need:

- is extracted from one or more raw data artifacts

- is written in user language

- is explicitly **linked back to its source artifact** using the **traced to** relationship on innoslate

This document represents the **output of the empathizing stage** of design thinking.

### **Why this deliverable is important**

- It prevents teams from designing based on assumptions.

- It establishes **traceability from real users to formal requirements**.

- It protects the project from later scope disputes (“this was never asked for”).

- Weak user needs lead to weak requirements — and failed systems.

If a need cannot be traced to a real source, it **does not belong** in this document.

### **Guidelines**

The important information previously captured as raw Statement entities in the Notes Documents (from multiple evidence sources) must now be consolidated into a single **User Needs Document**.

Identify statements in the Notes Document that are relevant for specifying user needs. For each raw Statement identified:

- If the statement is poorly phrased or ambiguous, create a new well-formed Statement.

- Otherwise, preferably, reuse the existing raw Statement by adding it directly to the User Needs Document.

<u>In both cases, create a new statement in the User Needs Document “traced to” the statements in the Notes Document.</u>

Below are the steps to **create a User Needs Document** and **add Statement entities to the document**:

1.  Create User Needs Document

    1.  Go to “Documents” View and create a new “User Needs Document”.

    2.  Name the Document as “User Needs Document”.

    3.  Number the Document with “UN.1” and add a description.

2.  Add statement entity

    1.  Add new Statement entity

        1.  Click on the ![](../../assets/images/image11.png){ style="width:0.53646in;height:0.25876in" } button to create a new Statement entity.

        2.  Number the statement as ”UN.1.n”.

        3.  Specify the relationship between statements using “traced to”.

3.  Create RTM (xlsx) report from user needs document and add it to the submission report document. (copy paste the table)

**Student Checklist – User Needs Document**

#### **1. User Needs Document Creation**

☐ Only **one consolidated User Needs Document** exists for the project  
☐ Document number is **UN.1  
** ☐ Document contains a meaningful description explaining scope and stakeholders

#### **2. Selecting Statements from Notes Documents**

☐ Only statements relevant to **user needs** are selected from Notes Documents  
☐ Every need originates from previously captured evidence (EX statements)  
☐ Sources refer to specific evidence (interview, visit, recording, reference), not vague sources like “discussion”

#### **3. Reusing vs Creating Statements**

For each identified need:

**If the raw statement is clear:  
** ☐ Existing Statement entity is added using **Add Existing Entity  
** ☐ Statement is renumbered to **UN.1.n**

**If the raw statement is unclear/ambiguous:  
** ☐ A new well-formed Statement entity is created  
☐ Statement is numbered **UN.1.n  
** ☐ The new statement is linked to the original EX statement using **“traced to”**

#### **4. Traceability**

☐ Every UN statement is traced to at least one EX statement  
☐ “traced to” relationships exist and are correctly assigned  
☐ No user need exists without a source statement  
☐ Multiple sources are linked when a need is supported by multiple evidence items

#### **5. Quality of Needs**

☐ Each need is one atomic statement (not combined requirements)  
☐ Written in user language (what the user must be able to do), not implementation language  
☐ No solutions, features, or design decisions included  
☐ No duplicated or overlapping needs

#### **6. Consistency & Numbering**

☐ All needs follow sequential numbering **UN.1.1, UN.1.2, UN.1.3…  
** ☐ No EX numbering remains inside the User Needs Document  
☐ All statements inside the document are Statement entities (not plain text)
